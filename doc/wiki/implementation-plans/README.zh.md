# 实现计划规范

> 状态：初稿

本目录存储具体的工程计划，用于构建设计页面和用例所描述的变更。

实现计划描述了工作应如何在代码中完成。

## 文件命名

使用格式：

```text
MM-NNNN-short-name.md
```

- `MM`: 来自 [../module-ids.zh.md](../module-ids.zh.md) 的模块/领域 ID
- `NNNN`: 四位十进制实现计划编号
- `short-name`: 以连字符连接的小写单词

示例：`00-0001-logging-importer.md`

使用来自 [../wiki-conventions.zh.md](../wiki-conventions.zh.md) 的规范/草稿工作流和状态术语。

## 模板

```md
# IP MM-NNNN: 简短标题

> 状态：草稿 (draft)
> 模块：模块名称
> 相关设计：../relative/path/to/design.zh.md
> 相关用例：../relative/path/to/use-case.zh.md

## 目标

用一两句具体的话描述实现结果。

## 范围

- 本计划将要实现的内容。

## 非目标

- 本计划有意不实现的内容。

## 输入与输出

- 所需的输入、配置、文件、数据库表或 API。
- 预期的输出、副作用或持久化数据。

## 文件与模块

- 预计创建或更改的文件或模块。
- 如果工作可能被委派，则说明所有权边界。

## 数据流

1. 主要处理步骤。
2. 主要处理步骤。
3. 主要处理步骤。

## 错误处理

- 预期的错误案例以及实现应如何报告或恢复。

## 验证

- 证明计划成功的命令、检查项、样本数据或手动评审步骤。

## 工作分解

- 可以分配给人类或代理的小型实现任务。

## 开放性问题

- 在实现之前或期间必须回答的问题。
```

## 编写规则

- 保持计划足够具体，以指导编码。
- 链接到相关的设计页面和用例，而不是重复它们。
- 当文件和模块已知时，请指明它们。
- 记录非目标以防止范围蔓延。
- 偏好明确的数据流和错误处理，而非模糊的实现说明。
- 当独立贡献者可以分别负责不同部分时，请拆分大型计划。
- 如果多个代理或人类可能并行工作，请清晰标注委派边界。
