# 遗留子系统

> 状态：初稿

## 摘要

遗留系统被划分为若干面向用户的访问层、服务守护进程、共享库和辅助实用程序。其中一些子系统提供直接的用户访问，而另一些则主要用于支持遗留运行时模型或共享通用逻辑。本页面旨在从高层描述主要的子系统边界，而非记录详细的代码路径。它是当前系统形态的一份图谱，后续可以通过代码阅读和迁移规划进行细化。

## 子系统列表

本页面涵盖的主要遗留子系统包括：

- `bbsd`
- `bbs`
- `smth_sshbbsd`
- `nju09/www`
- `api`
- `web`
- 支持库，如 `libythtbbs`、`libytht` 和 `libbmy`

## 当前角色

### `bbsd`

- 主要角色：
  - Telnet 守护进程，接受传入的终端连接并启动 `bbs`。
- 访问路径：
  - 发送到配置好的 BBS 端口的 Telnet 请求。
- 备注：
  - 它在多个端口上进行监听。
  - 其中一条路径支持 BIG5 模式，但主要执行流是相似的。

### `bbs`

- 主要角色：
  - 主要的遗留服务二进制文件，实现了大部分面向终端的 BBS 行为。
- 访问路径：
  - 由 `bbsd` 生成，也通过 `smth_sshbbsd` 间接进入。
- 备注：
  - 它是许多遗留用户和管理工作流的核心实现。

### `smth_sshbbsd`

- 主要角色：
  - SSH 守护进程，以及进入 `bbs` 执行流的基于 SSH 的入口点。
- 访问路径：
  - 发送到 `22` 端口的 SSH 请求。
- 备注：
  - 一个进程监听传入的 SSH 连接。
  - 子进程为每个连接提供服务，然后过渡到系统的 `bbs` 侧。
  - 它将 SSH 1.x 代码与 `bbs` 源代码相结合，因此其流程与 `bbs` 相关但并不完全相同。

### `nju09/www`

- 主要角色：
  - 基于 CGI 的遗留 Web 界面。
- 访问路径：
  - 由外部 Web 服务器路由到 CGI 入口点的 HTTP 请求。
- 备注：
  - Web 服务器处理到 CGI 二进制文件的路由。
  - 然后 CGI 层解析请求并分发到匹配的函数。
  - 该界面提供浏览器访问，但比终端界面更受限。

### `api`

- 主要角色：
  - 实验性的基于 HTTP 的 API 层。
- 访问路径：
  - 在其自身的端口上监听，预计由 HTTP 服务器进行反向代理。
- 备注：
  - 它提供了访问 BBS 功能的另一种路径。
  - 目前主要与 `web` 配合使用。

### `web`

- 主要角色：
  - 旨在与 `api` 配合使用的现代 Web 前端。
- 访问路径：
  - 由 HTTP 服务器作为面向浏览器的前端提供服务。
- 备注：
  - 它不是一个独立的后端。
  - 相比遗留的 CGI 层，它更能代表未来面向浏览器的客户端方向。

### 支持库或共享层

- `libythtbbs`:
  - 提供与 BBS 流程相关的核心数据结构和共享函数。
  - 被 `bbs`、`nju09/www` 以及 `local_utl` 下的实用程序使用。
- `libytht`:
  - 提供不与特定 BBS 流程紧密绑定的底层助手函数。
  - 示例包括字符串操作、文件操作、随机数实用程序、加密助手、时间函数和其他通用助手。
- `libbmy`:
  - 提供并非来自上游 YTHT 项目的项目特定助手。
  - 示例包括 Cookie 管理、MySQL 封装、SMTP 支持和 2FA 相关函数。
- `include`:
  - 收集这些库和相关代码使用的共享头文件。

## 已知关系

从高层来看，当前的子系统关系如下：

- `bbsd` 为基于 Telnet 的访问启动 `bbs`。
- `smth_sshbbsd` 嵌入或过渡到 `bbs` 逻辑以进行基于 SSH 的访问。
- `nju09/www` 为相似或重叠的系统行为提供了一条独立的基于 CGI 的 Web 路径。
- `api` 提供了一条较新的基于 HTTP 的访问路径，目前主要与 `web` 配合使用。
- 遗留系统并非纯粹基于文件系统；一些较新的或辅助的部分也依赖于基于 MySQL 的存储路径。
- `web` 依赖于 `api` 而非遗留的 CGI 层。
- `libythtbbs`、`libytht` 和 `libbmy` 等共享库支持整个仓库中的多个可执行文件和实用程序。

## 可能的迁移计划

基于目前的迁移方向，这些子系统并非都有着相同的未来。

- `bbsd`、`bbs`、`smth_sshbbsd` 和 `nju09/www` 预计将作为长期运行时组件退役。
- `api` 并非最终的目标架构，但相比旧的访问层，它更接近未来面向 API 的方向。
- `web` 更有可能演进而非立即退役，尽管其确切的未来形态仍是开放的。
- `libythtbbs`、`libytht` 和 `libbmy` 应被谨慎对待：某些部分在迁移过程中可能仍然有用，某些部分可能会被替换，而某些部分可能仅作为参考。
- 即使不再作为长期的运行时基础，整个遗留代码库在迁移过程中仍然是重要的参考。
