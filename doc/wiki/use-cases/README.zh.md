# 用例规范

> 状态：初稿

本目录存储具体的行为场景，作为设计页面与实现工作之间的桥梁。

用例描述了**必须**实现的功能。它们不应描述代码将如何构建。

## 文件命名

使用格式：

```text
MM-NNNN-short-name.md
```

- `MM`: 来自 [../module-ids.zh.md](../module-ids.zh.md) 的模块/领域 ID
- `NNNN`: 四位十进制用例编号
- `short-name`: 以连字符连接的小写单词

示例：`00-0001-import-one-daily-log.md`

使用来自 [../wiki-conventions.zh.md](../wiki-conventions.zh.md) 的规范/草稿工作流和状态术语。

## 模板

```md
# UC MM-NNNN: 简短标题

> 状态：草稿 (draft)
> 模块：模块名称
> 相关设计：../relative/path/to/design.zh.md

## 目标

用一两句具体的话描述该场景。

## 前置条件

- 所需的数据、配置、模式 (schema)、服务或环境状态。

## 基本流程

1. 第一个正常步骤。
2. 第二个正常步骤。
3. 最后的正常步骤。

## 备选流程

- 重要的分支、错误路径、重试路径或跳过的路径。

## 后置条件

- 流程完成后预期的状态。

## 验证

- 可以证明该用例正常工作的具体检查项。

## 备注

- 可选的上下文、不确定性或实现提示。
```

## 编写规则

- 保持用例具体且基于场景。
- 偏好可观察的行为，而非内部实现细节。
- 链接到相关的设计页面，而不是重复它们。
- 明确记录假设。
- 当备选流程影响实现或验证时，请包含它们。
- 保持验证步骤足够实用，以便人类或代理可以将其转化为检查。
