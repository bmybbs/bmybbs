# 迁移阶段

> 状态：初稿

## 摘要

迁移应分阶段进行，而非通过一次性重写。系统规模巨大，目标技术栈引入了新的基础设施和工具，并且一些关键的技术方向仍需在实践中验证。在项目深入到耦合度更高且风险更大的系统部分之前，每个阶段都应验证一组有限的假设。

## 总体策略

- 为何要分阶段迁移：
  - 系统过于庞大且交织紧密，无法通过一个步骤安全迁移。
  - 团队仍在积累 PostgreSQL 和未来技术栈的经验。
  - 早期阶段应在项目触及更核心的面向用户的数据和工作流之前减少不确定性。
- 每个阶段应验证什么：
  - 所选的技术方向在实践中是否可行
  - 团队是否能可靠地操作新的存储和迁移工作流
  - 遗留组件与新组件之间的边界是否可行
  - 在需要时迁移路径是否保持可逆
- 应避免什么：
  - 尝试首先迁移耦合度最高或风险最大的子系统
  - 将探索性实现视为已定义的最终架构
  - 强迫所有数据域和客户端路径同时移动

## 第一阶段：日志系统迁移

- 目标：
  - 将遗留的日志文本移动到 PostgreSQL 中，并构建辅助路径以便新日志也能写入数据库。
- 范围：
  - 仅限日志相关数据。
  - 暂不尝试迁移核心面向用户的内容（如版面或文章）。
- 为何首先进行此阶段：
  - 与文章和面向用户的工作流相比，日志记录与核心产品行为的耦合度较低。
  - 它是验证数据库方向的较低风险子系统。
- 验证内容：
  - PostgreSQL 在该项目中的集成情况
  - 初始模式 (schema) 设计习惯
  - 迁移工具模式
  - 遗留的基于磁盘的数据与新的数据库后端数据之间的共存
  - 已迁移数据域的备份和存储规划
- 退出准则：
  - 现有的日志数据可以迁移到 PostgreSQL
  - 可以通过稳定的辅助路径将新日志记录到 PostgreSQL 中
  - 团队有足够的信心将类似模式应用于其他低风险数据域
- 备注：
  - 本阶段旨在证明可行性，而非定义整个未来的数据架构。

## 第二阶段：结构化数据与 MySQL 退役

- 目标：
  - 将相对解耦的结构化数据迁移到 PostgreSQL，并从目标运行时中退役 MySQL。
- 范围：
  - 用户数据
  - 版面数据
  - 订阅相关数据
  - 文章元数据
  - 目前存储在 MySQL 中的注册数据
- 候选子系统或数据组：
  - 这些数据域被视为一个相关联的组，因为它们是结构化的、联系紧密的，并且已经通过早期基于 MySQL 的实验积累了一些迁移历史。
  - 文章元数据属于本阶段，因为它是结构化数据且被订阅相关行为所使用。
- 为何本阶段是合适的：
  - 它将数据库迁移方法从日志扩展到更重要但仍可控的结构化数据。
  - 该领域的一些部分已在之前的实验中进行过探索，这减少了不确定性。
  - MySQL 目前持有属于本阶段的结构化数据，以及注册数据。
- 验证内容：
  - 更多结构化关系数据的迁移
  - 与剩余遗留存储的共存
  - 用基于 PostgreSQL 的等效项替代基于 MySQL 的部分
  - 新的模式方向是否能扩展到日志实验之外
- 退出准则：
  - 用户、版面、订阅、文章元数据和注册数据可以可靠地通过基于 PostgreSQL 的路径进行存储和读取
  - 所有当前基于 MySQL 的数据都有了基于 PostgreSQL 的替代方案
  - 验证后可以从目标运行时退役 MySQL
  - 团队对于关系模式演进和迁移工具拥有了更清晰的模型
- 备注：
  - 本阶段仍避开了文章正文、附件负载、邮件内容和内容编码的全部复杂性。

## 第三阶段：文章内容与附件迁移

- 目标：
  - 将文章正文迁移到新的存储模型中，并将附件负载分离到对象存储中。
- 范围：
  - 文章正文
  - 附件元数据
  - 附件负载存储
- 候选子系统或数据域：
  - 文章内容和附件存储
- 验证内容：
  - 处理系统中最重要的内容领域
  - 实际迁移内容中 GBK 与 UTF-8 的共存
  - 附件引用处理与存储分离
  - 规划中的 PostgreSQL 加对象存储模型是否适用于核心内容
- 退出准则：
  - 文章正文和附件可以通过新的存储架构进行存储和访问
  - 核心内容的迁移设计足够稳定，能够支持后续以 API 为中心的工作
- 备注：
  - 文章元数据预计已在第二阶段处理。
  - 这预计将是风险最高的阶段之一，因为它涉及核心内容负载模型。

## 第四阶段：邮件迁移

- 目标：
  - 在文章内容和附件迁移验证了核心内容存储方法之后，迁移邮件数据。
- 范围：
  - 邮件元数据
  - 邮件内容
  - 邮件相关附件处理（如适用）
- 候选子系统或数据域：
  - 站内信及相关的个人消息存储
- 验证内容：
  - 将文章迁移模式重用于类似但独立的用户数据域
  - 共享的类文章数据结构是否能干净地适配邮件
  - 个人/私有内容的迁移是否引入了额外的运维或访问控制顾虑
- 退出准则：
  - 邮件数据可以通过新的存储架构进行存储和访问
  - 项目对于迁移公共版面内容之外的类文章领域有了更清晰的模型
- 备注：
  - 邮件与文章迁移分离是因为它是一个独立的用户数据域，即使它共享类似的数据结构。

## 第五阶段：以 API 为中心的功能迁移

- 目标：
  - 将 API 层转向直接消费已迁移的数据，而非依赖遗留的面向 GBK 的原始存储和转换包装器。
- 范围：
  - 面向 API 的数据访问
  - 面向 UTF-8 的内容交付
  - 可以开始使用新存储路径的选定功能
- 候选子系统或功能组：
  - API 数据访问和早期的 API 后端功能路径
- 验证内容：
  - 新的数据模型是否能支持直接的 API 消费
  - 以 UTF-8 为先的访问路径是否简化了当前的 API 行为
  - 功能迁移是否能增量进行而非通过全量切换
- 退出准则：
  - 一些 API 功能运行在已迁移的存储路径上，而不依赖于旧的原始内容访问模式
  - 项目在遗留访问层与新的以 API 为中心的行为之间有了更清晰的边界
- 备注：
  - 确切的 API 框架或语言栈在此阶段或之后仍可能演进。

## 第六阶段：客户端过渡与遗留退役

- 目标：
  - 逆向工程 `bbs` 和 `nju09/www` 的主要流程，通过 API 层重新实现保留的行为，并继续从遗留客户端和守护进程过渡。
- 范围：
  - 主要面向用户的流程
  - 选定保留的管理相关流程
  - 跨 Web 和未来 TUI 路径的客户端过渡工作
- 候选子系统或功能组：
  - `bbs`
  - `nju09/www`
  - 未来由 API 支持的 Web 和 TUI 行为
- 验证内容：
  - API 层是否能承载保留的核心工作流
  - 未来的客户端是否能取代重要的遗留访问模式
  - 项目是否能在不丢失必要功能的情况下退役主要的遗留运行时组件
- 退出准则：
  - 新架构中存在重要的保留流程
  - 遗留运行时可以开始真正的退役而非仅仅共存
- 备注：
  - 到此阶段，API 实现栈的部分内容可能也已演进到早期的实验形态之外。

## 阶段风险

- 低风险迁移领域的早期成功可能会对更困难的内容和工作流迁移产生盲目自信
- 遗留存储与新存储之间的过渡性共存持续时间可能长于预期
- 过早做出的模式或存储决策在核心内容迁移开始时可能需要修订
- 在第二阶段之后退役 MySQL 取决于确认注册数据及所有其他基于 MySQL 的领域都有了基于 PostgreSQL 的替代方案

## 开放性问题

- 是否应专门为后台任务和维护实用程序预留一个未来的阶段，或者应将这些内容合并到周边的数据库和 API 迁移阶段中？
- 哪个阶段应承担对未来 TUI 方向的第一次严肃验证？
