# 日志系统第一阶段：接口重构

> 状态：初稿

## 摘要

第一阶段是一个保持存储方式不变的重构。它用语义化的 `bmy_log_*` 接口取代直接的 `newtrace` 调用，同时保持现有的运行时路径不变。

## 范围

- 用按事件家族分组的、更清晰的日志接口取代直接调用 `newtrace` 的行为。
- 保持当前的存储路径不变：
  - 调用者最终仍将文本发送到 `newtrace`
  - `newtrace` 仍通过 SysV 队列发送消息
  - `bbslogd` 仍在地盘上写入每日日志文件
- 本阶段不引入 PostgreSQL 写入。
- 本阶段不导入现有的历史日志文件。

## 目标

目标是使写入路径变得足够明确，以便后续的迁移工作可以清晰地看到实践中存在哪些类型的日志类别。

在本阶段之后，项目应对以下内容有更清晰的了解：

- 哪些日志是面向业务的
- 哪些日志是与仲裁或审计相关的
- 哪些日志是面向统计的
- 哪些日志是仅限运维或调试的

## 目标行为

- 每个调用者都应使用语义化接口，而不是直接调用 `newtrace`。
- 这些接口背后的实现应仍向 `newtrace` 发出相同的文本消息。
- 磁盘上的日志格式、文件布局和运维工作流应保持不变。
- 检查当前日志文件的现有工具应能继续工作。

## 事件家族

- 账号与会话事件
- 版面使用与统计事件
- 发帖、邮件与内容生命周期事件
- 仲裁与行政内容状态事件
- 运维与集成诊断

这里的目的还不是重新设计存储，而是强迫直接使用 `newtrace` 的行为进入更清晰的类别。

## 当前状态

语义化日志层现在位于：

- [include/bmy/logging.h](../../../../include/bmy/logging.h)
- [libbmy/logging.c](../../../../libbmy/logging.c)

当前的实现仍会格式化兼容遗留系统的文本并调用 `newtrace`。

## 退出准则

- 直接调用 `newtrace` 的行为被语义化日志接口所取代。
- 对于现有的日志行为，发出的文本保持不变。
- 项目有了一个更清晰、可评审的日志类别图谱，用以指导第二阶段的数据库设计。
