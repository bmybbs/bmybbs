# 日志系统第二阶段：数据库设计与历史导入

> 状态：初稿

## 摘要

第二阶段证明遗留的基于文件的 `newtrace` 日志可以转换为可查询的 PostgreSQL 数据。

本阶段有两条工作线：

- 数据库设计：对语义化日志 API 进行分类，并定义稳定的 PostgreSQL 表
- 历史导入：构建一个导入器，用于读取现有的每日日志文件并写入分类后的行

第二阶段仍应避免从活跃代码中直接进行生产数据库写入。直接实时写入属于第三阶段。

## 详细页面

- 数据库设计：[phase-2-database-design.zh.md](./phase-2-database-design.zh.md)
  - 类别表、丢弃的 API、导入追踪和索引的规范数据库设计。
  - 可执行的 SQL 资产为 [db/bmy.pg.sql](../../../../db/bmy.pg.sql)。
- 导入器设计：[phase-2-importer.zh.md](./phase-2-importer.zh.md)
  - 命令行形态、解析边界、事务、编码和验证的规范导入器设计。

## 当前状态

- 初始 PostgreSQL 模式已在 [db/bmy.pg.sql](../../../../db/bmy.pg.sql) 中实现。
- `local_utl/log_importer` 下的 C 语言导入器实现了单日导入、试运行 (dry-run) 解析、分类插入、导入追踪和摘要计数器。
- 导入器在 `libbmy` 中使用了一个精简的 PostgreSQL 封装器，并已集成到 CMake 构建定义中。
- 针对测试环境和历史日志文件的运行时验证仍在进行中。

## 边界

- 在第二阶段，保持 `bbslogd` 和现有的磁盘日志作为活跃的生产路径。
- 保留原始日志文件作为原始备份源。
- 仅导入分类后的业务/事件日志。
- 不要将丢弃的运行时诊断信息导入业务事件表。
- 不要向日志表添加外键。

## 待办工作

- 针对历史日志运行试运行 (dry-run) 扫描，以发现较旧或意外的格式。
- 使用测试数据验证导入幂等性和选定的事件家族映射。
