# 日志系统第二阶段：导入器设计

> 状态：初稿

## 摘要

本页面记录了用于将遗留 `newtrace` 日志导入 PostgreSQL 的导入器设计。

导入器旨在提供一种可靠、幂等的方式，将历史磁盘日志转换为结构化的数据库记录。

## 设计原则

- 优先考虑幂等性。
- 分离解析器逻辑与导入器外壳。
- 使用事务确保数据完整性。
- 提供试运行 (dry-run) 模式，用于在不写入数据库的情况下验证解析。
- 保持摘要计数器稳定且有意义。
- 允许稍后在不更改导入器核心逻辑的情况下支持更多的日志类别。

## 导入器外壳

导入器外壳负责 CLI 交互、文件解析、数据库连接、事务管理以及摘要报告。

### 命令行形态

```text
log_importer [--dry-run] YYYY-MM-DD
```

- `YYYY-MM-DD`: 必需的日期参数。用于定位原始日志文件并重建事件时间戳。
- `--dry-run`: 可选。解析并报告摘要，但不检查导入追踪也不插入行。

### 文件定位

导入器通过日期参数解析来源文件：

- 默认位置：`$HOME/newtrace/YYYY-MM-DD.log`
- 存储在 `log_imported_lines.source_file` 中的规范名称仅为 `YYYY-MM-DD.log`。

### 处理流程

1. 解析命令行参数。
2. 验证日期参数。
3. 打开来源日志文件。
4. 逐行读取文件内容。
5. 对于每一行：
   - 检查 `log_imported_lines` 以确保该物理行尚未被导入（试运行模式除外）。
   - 如果尚未导入，则调用解析器。
   - 如果解析器识别出已接受的事件，则：
     - 使用日期参数和行内时间重建 `occurred_at`（默认为 UTC+8）。
     - 将事件插入匹配的类别表。
     - 在 `log_imported_lines` 中记录该物理行已被导入。
     - 在同一个事务中提交这两次插入。
6. 输出包含计数器的最终摘要。

## 解析器组件

解析器负责将单行文本分类并转换为类型化的事件数据。

### 职责

- 提取行内时间（`HH:MM:SS`）。
- 将行分类为：已接受、已丢弃、未识别或解析失败。
- 对于已接受的事件，返回目标表标识符和类型化的列值。
- 使用 `g2u` 将遗留的自由文本字段从 GBK 转换为 UTF-8。
- 保留 `*` 作为普通内容。

### 分类逻辑

- **已接受 (Accepted)**: 匹配已定义类别表的已知业务或审计事件。
- **已丢弃 (Discarded)**: 匹配已知但不需要导入数据库的运行时诊断或追踪 API。
- **未识别 (Unrecognized)**: 不匹配任何已知模式的行。
- **解析失败 (Failed)**: 看起来像已接受的事件，但包含无效的字段计数、无效的数字或无法转换编码。

## 编码与转换

- 遗留日志中的自由文本（如文章标题）采用 GBK 编码。
- 导入器在将这些字段存入 PostgreSQL 之前，必须使用 `convcode.h` 中的 `g2u` 助手将其转换为 UTF-8。
- 如果转换失败，该行应计为“解析失败”。

## 事务策略

- 类别表插入和 `log_imported_lines` 插入必须是原子的。
- 初始实现：每条导入的事件行使用一个事务。
- 理由：
  - 这确保了历史导入期间的失败隔离。
  - 简化了断点重试逻辑。
  - 如果导入速度成为瓶颈，以后可以考虑分批优化。

## 幂等性与重复导入

导入器通过以下方式避免重复：

- 检查 `log_imported_lines` 中是否存在 `(source_file, source_line)` 的组合。
- 在插入事件的同时原子化地记录该行。
- 试运行 (dry-run) 模式跳过此检查。

## 验证策略

- **单元测试**: 解析器组件应针对样本行具有独立的单元测试。
- **试运行扫描**: 针对旧的历史日志运行试运行模式，以发现未识别或失败的模式。
- **样本导入**: 导入包含已知事件类型的测试日志并验证数据库结果。
- **幂等性测试**: 对同一个日期运行两次导入，确认第二次运行报告“已导入”计数且没有产生重复行。

## 局限性

- 初始版本硬编码为 UTC+8 时区处理。
- 初始版本不支持在一个进程中批量导入多个日期。
- 初始版本不提供 JSON 摘要输出。
- 解析器匹配逻辑基于当前对日志模式的理解；历史格式漂移（format drift）是预期的风险。
