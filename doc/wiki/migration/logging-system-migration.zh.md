# 日志系统迁移

> 状态：初稿

## 摘要

本页面是遗留日志系统分阶段迁移的概览。长期目标是从当前基于文件的 `newtrace` 和 `bbslogd` 管道转向基于数据库的业务日志，同时保持迁移过程足够细化，以便按阶段进行评审。

日志记录是第一个迁移实验，因为与版面、文章或邮件存储相比，它与核心面向用户行为的耦合度较低。它也是将业务事件与运维诊断分离的一个有用案例研究。

## 阶段概览

### 第一阶段：接口重构

详细页面：[logging-system/phase-1-refactor.zh.md](./logging-system/phase-1-refactor.zh.md)

第一阶段用语义化的 `bmy_log_*` 接口取代直接的 `newtrace` 调用，同时保持现有的基于文件的存储路径不变。

当前状态：

- 语义化接口位于 [include/bmy/logging.h](../../../include/bmy/logging.h)
- 兼容性实现位于 [libbmy/logging.c](../../../libbmy/logging.c)
- 存储仍流经 `newtrace`、SysV 队列、`bbslogd` 和每日磁盘文件

### 第二阶段：数据库设计与历史导入

详细页面：[logging-system/phase-2-import.zh.md](./logging-system/phase-2-import.zh.md)

第二阶段在第一阶段理清日志类别后设计数据库结构。它还为现有的基于文件的 `newtrace` 日志引入历史导入功能。在此阶段仍应避免从活跃代码中直接进行生产数据库写入。

当前预览：

- 按类别逐步设计模式
- 为稳定的事件组提供专用表
- 在类别设计完成前，保持未建模事件在导入范围之外
- 保持导入追踪与类别表分离
- 为现有的基于文件的日志引入日志导入器
- 每次调用接受一个日志文件路径并仅处理该文件
- 以后支持通过脚本驱动批量导入，而非一次性的单体导入
- 在直接数据库写入部署到生产环境之前，保持导入器在过渡期内持续可用

### 第三阶段：直接数据库写入

详细页面：[logging-system/phase-3-direct-write.zh.md](./logging-system/phase-3-direct-write.zh.md)

第三阶段将生产环境的写入路径从仅限文件的日志记录转向直接数据库写入。在第二阶段验证模式和导入器方案之前，本阶段有意保持简略。

## 导入与共存策略

- 第一阶段：基于文件的日志记录仍是唯一的活跃存储路径。
- 第二阶段：基于文件的历史日志通过单文件导入器逐步导入。
- 第三阶段：引入直接数据库写入，同时导入工具在切换稳定前保持可用。

## 当前评估

- 第一阶段已经产生了后期模式设计所需的语义化日志边界。
- 第二阶段仍保持为数据库设计和历史导入阶段，并按事件类别增量设计模式。
- 运行时诊断最终可能会使用独立的服务日志方案，而非业务日志模式。
