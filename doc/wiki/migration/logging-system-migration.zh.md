# 日志系统迁移

> 状态：已播种 (seeded)

## 摘要

本页面描述了遗留日志系统的分阶段迁移方案。长期目标是从当前基于文件的 `newtrace` 和 `bbslogd` 管道转向基于数据库的日志系统，但第一阶段暂不涉及数据库。第一阶段是一个重构阶段，其目的是使现有的日志行为更易于理解、分类，并为以后的安全迁移做准备。

## 为何日志系统先行

- 与版面、文章或邮件存储相比，日志系统与核心面向用户行为的耦合度较低。
- 在触及更核心的数据领域之前，这是一个验证迁移工作流和重构纪律的好地方。
- 当前的日志路径已经包含了几种不同的事件家族，因此它是将业务事件与运维诊断分离的一个有用案例研究。

## 第一阶段：不改变存储的接口重构

### 范围

- 用按事件家族分组的、更清晰且有意义的日志接口取代直接调用 `newtrace` 的行为。
- 保持当前的存储路径不变：
  - 调用者最终仍将文本发送到 `newtrace`
  - `newtrace` 仍通过 SysV 队列发送消息
  - `bbslogd` 仍在地盘上写入每日日志文件
- 本阶段不引入 PostgreSQL 写入。
- 本阶段不导入现有的历史日志文件。

### 目标

第一阶段的主要目标是使写入路径变得足够明确，以便后续的迁移工作可以清晰地看到实践中存在哪些类型的日志类别。在本阶段之后，项目应对以下内容有更清晰的了解：

- 哪些日志是面向业务的
- 哪些日志是与仲裁或审计相关的
- 哪些日志是面向统计的
- 哪些日志是仅限运维或调试的

### 目标行为

- 每个调用者都应使用语义化接口，而不是直接调用 `newtrace`。
- 在第一阶段，这些接口背后的实现应仍向 `newtrace` 发出完全相同的文本消息。
- 磁盘上的日志格式、文件布局和运维工作流应保持不变。
- 检查当前日志文件的现有工具应能继续工作。

### 首先要重构的事件家族

- 账号与会话事件
- 版面使用与统计事件
- 发帖、邮件与内容生命周期事件
- 仲裁与行政内容状态事件
- 运维与集成诊断

这里的目的还不是重新设计存储，而是强迫每个直接的 `newtrace` 调用进入一个更清晰的类别。

### 验证目标

- 重构后的代码应仍能产生与之前相同的日志文本。
- 直接调用 `newtrace` 的行为应从子系统代码中消失，仅保留在新增的接口背后。
- 新的接口布局应使未来的数据库模式设计更容易，因为事件家族比以前更清晰了。

### 退出准则

- 直接调用 `newtrace` 的行为被新的语义化日志接口所取代。
- 对于现有的日志行为，发出的文本保持不变。
- 项目有了一个更清晰、可评审的日志类别图谱，用以指导下一阶段的数据库设计。

## 第二阶段预览：数据库设计与历史导入

- 在第一阶段理清日志类别后，设计数据库结构。
- 为现有的基于文件的日志引入日志导入器。
- 导入器应在每次调用时接受一个日志文件路径，并仅处理该文件。
- 批量导入应在以后通过重复调用导入器的脚本完成，而非通过一次性的单体导入任务。
- 在直接数据库写入部署到生产环境之前，导入器在过渡期内应保持有用。

## 第三阶段预览：直接数据库写入

- 将生产环境的写入路径从仅限文件的日志记录转向直接数据库写入。
- 在部署新路径的同时，保持导入器可用于历史补全、回放或切换支持。
- 使用第一阶段的接口边界和第二阶段的模式/导入经验作为最终写入路径转换的基础。

## 导入与共存策略

- 第一阶段：基于文件的日志记录仍是唯一的活跃存储路径。
- 第二阶段：基于文件的历史日志通过单文件导入器逐步导入。
- 第三阶段：引入直接数据库写入，同时导入工具在切换稳定前保持可用。

## 当前评估

- 第一阶段是有意采取的保守方案。
- 它尚未验证 PostgreSQL 的集成。
- 相反，它减少了写入路径中的歧义，使得下一阶段的数据库工作是基于更清晰的事件类别，而非原始的 `newtrace` 调用点。

## 开放性问题

- 第一阶段应引入多少语义化日志接口，其粒度应有多粗或多细？
- 在基于数据库的日志记录存在后，是否有些事件家族应保持面向文件的形式？
- 在第二阶段或第三阶段的哪个时间点，`bbslogd` 应从一个活跃的业务日志接收器收缩为一个更小的仅限运维的角色？
