# 项目总览

> 状态：初稿

本系统是针对单一大学的校园社区平台。其核心目的是提供一个电子公告板系统 (BBS)，用户可以在其中：

- 注册、登录并查询用户信息
- 按分区浏览版面
- 浏览版面内的文章（主题或非主题模式）
- 阅读、发布和回复文章
- 向其他用户发送站内信和短消息

管理员和版主还可以管理配置并执行与管理相关的任务。

## 当前访问模式

当前系统通过多个访问层暴露：

- 通过 Telnet 和 SSH 进行终端访问
  - `bbsd` 和 `smth_sshbbsd` 作为守护进程服务器
  - `bbs` 提供主要的终端用户界面，并且仍承载着许多与管理相关的工作流
- 通过 HTTP 和 HTTPS 进行 Web 访问
  - `nju09/www` 是基于 CGI 的 Web 界面，其功能比终端界面更受限
  - `api` 是实验性的基于 JSON 的访问层，由现代 `web` 前端使用

虽然存在一些实验性或次要的客户端，但它们并非长期规划的核心。

## 为何遗留系统难以维护

遗留系统中积累了几种不同的维护负担。

- 架构与存储
  - 遗留系统使用的是混合存储模型，而非单一清晰的数据层。
  - 对于系统的大部分内容，文件系统仍是主要的存储方式。
  - 一些较新的或辅助的数据领域也使用了 MySQL。
  - 许多记录以直接从 C 结构体映射的二进制数据形式存储。
  - 系统严重依赖操作系统设施和底层文件管理，而非由数据库处理这些关注点。
  - 基于文件系统的数据与基于 MySQL 的数据并存也增加了迁移的复杂性。
- 耦合与重复
  - 类似的业务逻辑在不同的访问层中被多次实现。
  - 许多函数与特定接口绑定过紧，而不是被组织为可重用的领域逻辑。
- 数据格式约束
  - 文章采用 GBK 编码，而现代接口期望使用 UTF-8。
  - 附件被嵌入在文章内容中，这使得存储和迁移变得更加困难。
- 实现债
  - 一些底层代码路径对系统调用结果的检查不够仔细，或者检查错误。
  - 代码库中还包含了一些在当时看来合理但现在难以推理或扩展的历史规避方案 (work-arounds)。

## 目标方向

长期目标是退役遗留的访问架构，并代之以一个以单一 API 层为核心的、更简单、更具可维护性的系统。

目前的规划方向是：

- 保留核心社区工作流，特别是用户管理、版面查看和文章查看
- 将主要数据存储移至 PostgreSQL，并随着时间的推移逐步退役剩余的基于 MySQL 的部分
- 继续使用 Redis 进行缓存
- 可能会将二进制附件转移到基于对象存储的设计中
- 用一个面向 API 的后端取代多个遗留访问层
- 继续演进现代 Web 前端
- 探索一种新的 TUI 客户端，在不保留旧实现模型的情况下提供类似于 `bbs` 的感受

并非所有遗留特性都会被迁移。例如，基于 SSH 1.x 的 SSH 守护进程预计将退役，游戏相关模块也可能会被放弃。

确切的目标技术仍属于规划决策而非最终承诺。遗留代码库将继续作为参考并可能仍会接收关键的错误修复，但它不打算作为新功能的长期基础。

## 本 Wiki 的范围

本 Wiki 旨在支持从遗留系统到新架构的迁移。

其目标是：

- 捕捉对当前系统的稳定理解
- 将确认的事实与推论及开放性问题区分开来
- 记录预期的目标架构和迁移方向
- 帮助将遗留子系统和工作流映射到未来的对等实现中

随着 Wiki 的增长，详细的子系统行为、存储设计、编码问题和迁移规划应拆分为各自的页面。
