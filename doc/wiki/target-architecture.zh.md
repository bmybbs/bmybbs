# 目标架构

> 状态：初稿

## 摘要

未来的系统应以单 API 层为核心的模块化系统取代历史架构。主要目标是保留核心工作流，减少对底层操作系统机制的依赖，并使未来的开发比当前的遗留代码库更容易。目标架构应同时支持现代 Web 客户端和未来的 TUI 客户端，同时将这些客户端与核心业务逻辑分离。目前尚未决定每个细节，特别是关于 API 风格和 UI 设计，因此本页面记录的是方向和边界，而非最终的技术设计。未来的 API 和客户端模型将部分通过对遗留功能进行逆向工程，然后针对更具可维护性的系统进行重塑来设计。

## 目标

- 在成熟的组件和框架上构建未来系统，使项目能够专注于 BMYBBS 本身，而不是重新实现操作系统层面的关注点。
- 将存储、业务逻辑、访问控制和客户端展示分离到更清晰的层级。
- 保留核心工作流，特别是用户管理、版面查看和文章查看。
- 使新架构具有足够的灵活性，以便在不强制进行大规模重写的情况下吸收后续模块和修订后的决策。
- 通过将共享逻辑移至后端，减少各客户端之间的重复。

## 拟议的系统形态

未来系统预计将由几个主要部分组成，其边界比现有系统更清晰。

- API 层：
  - API 层应成为面向用户和管理相关功能的主要访问入口。
  - 它应涵盖目前分布在 `bbs`、`nju09/www` 和 `api` 中的重要功能。
  - 第一版不需要镜像每个功能。
- 数据存储：
  - 主要结构化数据应移至 PostgreSQL。
  - 存储设计应支持过渡期，在此期间遗留的基于文件系统的数据、基于 MySQL 的数据以及新的基于 PostgreSQL 的数据可能共存。
- 缓存层：
  - 缓存和共享状态应脱离当前面向共享内存的模型。
  - Redis 仍然是一个可能的选择，因为它已被使用并符合预期的方向。
- 附件存储：
  - 二进制附件应与主要关系数据存储分开管理。
  - 附件模型应支持安全引用跟踪、删除和完整性检查。
  - 附件模型不应依赖遗留的文章嵌入方式作为永久设计。
- Web 客户端：
  - 当前的 `web` 前端可能是未来面向浏览器的客户端的基础。
  - 确切的 UI 设计仍是开放的，不应阻碍后端架构决策。
- TUI 客户端：
  - 未来的 TUI 客户端旨在以更便携、更现代的形式提供部分 `bbs` 体验。
  - 其确切的交互设计仍处于探索阶段。
- 后台任务和维护组件：
  - 目前隐藏在遗留守护进程或脚本中的一些职责可能需要变成明确的后台任务、维护工具或迁移实用程序。
  - 可能的例子包括热帖或置顶贴生成、计划的用户自动解封操作，以及目前由 cron 式工作流驱动的统计或维护实用程序。

## 各层职责

### API 层

它应该做什么：

- 通过稳定的程序接口暴露核心系统功能
- 集中身份验证、授权和权限检查
- 封装目前在多个访问路径中重复的业务逻辑
- 以适合现代消费者的格式（可能是 JSON 优先）返回面向客户端的数据
- 支持增量交付，以便可以分阶段发布不同的功能集

它不应该做什么：

- 依赖于特定客户端的展示逻辑
- 在多个服务入口点重复底层存储逻辑
- 在没有明确授权检查的情况下暴露特权操作
- 与一种客户端实现或一种 UI 风格紧密耦合

### 客户端

客户端应该做什么：

- 专注于展示、交互流和客户端用户体验
- 调用 API 层而不是重新实现业务规则
- 将相同的后端功能适配到不同的使用环境（如浏览器和终端）

客户端不应该做什么：

- 携带自己的权限逻辑或领域规则副本
- 直接依赖遗留存储布局或底层系统接口
- 成为实现横向后端行为的主要场所

### 数据与存储层

属于这里的内容：

- 结构化持久数据，如用户、版面、文章、配置和操作记录
- 从遗留格式迁移到新表示形式期间需要的兼容性考虑
- 常用访问路径所需的查询模式和索引
- 附件元数据和引用，即使二进制负载存储在主数据库之外

该层可能需要针对编码、模式演变和数据迁移的过渡设计，而不是从旧到新的一次性转换。

在过渡期间，可能需要 GBK 和 UTF-8 表示形式共存。GBK 应保留为历史系统的原始兼容形式，而 UTF-8 成为新 API 和客户端的面向未来的表示形式。写路径可能需要同步这两种形式，直到遗留运行时完全退役。

## 预期的退役

长期架构预计将退役几个遗留运行时组件。

- 作为主要服务入口点的遗留 Telnet 和 SSH 访问守护进程
- 要求面向用户的服务器进程随会话死亡并在断开连接时丢失进行中状态的连接模型
- 基于 CGI 的 Web 访问层
- 主要为支持旧运行时假设而存在的遗留守护进程模式
- 在不再适合统一架构的情况下，旧的独立聊天相关服务模型
- 不属于未来产品方向的遗留游戏相关模块

退役 Telnet 和 SSH 访问模型是计划中的。Telnet 作为长期远程访问方法是不可接受的，因为它未加密，且现有的 SSH 实现与 SSH 1.x 紧密挂钩。更重要的是，旧的长连接模型将用户会话与一个服务器端进程耦合得太紧。未来系统应允许短期的请求/响应交互和适当的长连接通道（如 WebSocket），而不使客户端稳定性依赖于一个遗留终端会话进程。

遗留代码库可能仍作为参考保留，并在过渡期间接收关键修复，但它不是预期的长期运行时基础。

## 计划的技术方向

可能的事情：

- PostgreSQL 作为主要结构化数据存储
- 随着基于 PostgreSQL 的替代方案变得可用，逐步退役遗留的基于 MySQL 的部分
- Redis 作为主要缓存层
- 面向对象存储的二进制附件设计
- Rust 是新实现栈（至少是部分）的有力候选，特别是在更安全的底层系统工作或可重用的核心库有价值的地方
- 一个不直接依赖 `libythtbbs` 和 `libytht` 作为长期基础的后端架构

仍开放或探索中的事情：

- 未来 API 层的确切实现栈
- Rust 是否成为新后端的主要实现语言，或者首先在选定的较低层引入
- 未来 Web 前端的确切客户端架构
- 未来 TUI 客户端的最终设计和范围
- 具体的对象存储产品或部署模型

## 迁移原则

迁移应由一小组架构原则指导，而不是通过一次性重写。

- 在追求功能完整性之前，先保留重要数据和核心工作流
- 偏好模块化边界，以便新系统的各部分可以独立演进
- 允许分阶段迁移，而不是要求单一切换
- 在目标设计仍不确定时，保持决策的可逆性
- 记录重要的权衡和决策历史，以便后续迭代保持可理解性
- 每当成熟的基础设施组件可以接管该职责时，减少对遗留操作系统级机制的依赖
- 通过逆向工程和重新设计从遗留行为衍生出新的 API 行为，而不是盲目复制遗留接口

## 开放性问题

- 遗留运行时退役后，为了兼容性或历史备份目的，GBK 数据应继续保留多久？
- 哪些管理和行政功能必须出现在最早的 API 发布中？
- 未来的后端应该首先设计为一个可部署的服务，还是一个服务内部具有更清晰边界的多个内部模块？
- 什么样的附件生命周期模型最适合迁移、删除安全性和未来的可扩展性？
- 当前操作工作流的哪些部分在新架构中仍需要专用的后台服务？
